# v0.2 Testnet Settlement Loop

v0.2 moves zkpay from a model-only alpha toward a real Sui testnet payment loop:

```txt
PaymentIntent -> wallet transaction -> Sui testnet digest -> RPC receipt verification
```

The goal is narrow: a merchant can create a payment, a payer can submit a Sui
stablecoin transfer, and the merchant can verify the transaction before
fulfillment.

## SDK Surface

Install the public alpha package:

```bash
npm install zkpay-sh@next
```

Build a wallet-submittable Sui transfer transaction:

```ts
import { ZkpayClient } from "zkpay-sh";

const zkpay = new ZkpayClient();
const payment = zkpay.createPayment({
  amount: "20",
  coin: "USDC",
  receiver: "0x...",
  label: "API credits"
});

const { transaction, amountAtomic } = zkpay.buildSuiPaymentTransaction({
  intent: payment.intent,
  payer: "0x...",
  coinType: "0x...::usdc::USDC",
  decimals: 6,
  binding: {
    packageId: "0x..." // package exposing receipt::bind
  }
});
```

The returned `transaction` is a Sui TypeScript SDK `Transaction` that a wallet
adapter can sign and execute on testnet.

## Hosted Checkout Handoff

The hosted checkout route now loads a browser wallet runtime:

```txt
/pay/:paymentId?intent=...&network=testnet&coinType=0x...::usdc::USDC&decimals=6&bindingPackageId=0x...
```

SDK, API, and CLI payment creation can attach the Sui checkout parameters for
`network`, `coinType`, `decimals`, `rpcUrl`, `bindingPackageId`, and
`bindingEventType`. The page detects Wallet Standard Sui wallets, asks the
payer to connect, submits a stablecoin transfer, captures the returned
transaction digest, and then verifies receiver balance changes through Sui RPC.

After a digest is available, checkout renders the backend verification payload:

```json
{
  "intent": {},
  "txDigest": "H2j...",
  "coinType": "0x...::usdc::USDC",
  "decimals": 6,
  "expectedSender": "0x...",
  "binding": {
    "packageId": "0x..."
  },
  "amountPolicy": "exact",
  "options": {
    "enforceExpiration": true
  }
}
```

The wallet handoff is still an alpha settlement path. It does not custody funds
and it does not mark merchant orders paid by itself. Merchant backends must call
`/payments/verify/sui` or `verifySuiPayment` before fulfillment.

## Onchain Receipt Binding

v0.2 can bind the zkpay payment id and nonce into the submitted Sui transaction
through a small Move receipt event:

```txt
move/zkpay_receipt/sources/receipt.move
```

When a binding package id is supplied, the SDK appends a `receipt::bind` call to
the payment transaction. The call emits `PaymentBound` with payer, receiver,
amount, coin type, payment id, and nonce.

Because binding adds programmable transaction logic, apps should treat it as the
sponsored-gas route. The pure gasless stablecoin route remains best for simple
wallet-to-wallet transfers that do not need an onchain zkpay event.

```ts
const built = zkpay.buildSuiPaymentTransaction({
  intent: payment.intent,
  payer: "0x...",
  coinType: "0x...::usdc::USDC",
  decimals: 6,
  binding: {
    packageId: "0x..."
  }
});
```

Verification can then require that event:

```ts
const result = await zkpay.verifySuiPayment({
  intent: payment.intent,
  txDigest: "H2j...",
  coinType: "0x...::usdc::USDC",
  expectedSender: "0x...",
  binding: {
    packageId: "0x..."
  }
});
```

If the event is missing or does not match the payment intent, the verifier
returns `binding_event_missing` or `binding_event_mismatch`.

Build and publish the alpha package with:

```bash
npm run move:build
npm run move:publish:testnet
```

The publish script prints `ZKPAY_BINDING_PACKAGE_ID=...` after a successful
testnet publish. Use that package id in hosted checkout URLs, SDK checkout
options, and `/payments/verify/sui` binding config.

Use dry-run mode to verify the package path and Sui RPC access without spending
gas:

```bash
ZKPAY_RECEIPT_DRY_RUN=true npm run move:publish:testnet
```

Dry-run mode prints `ZKPAY_DRY_RUN_BINDING_PACKAGE_ID=...`; only real publishes
print `ZKPAY_BINDING_PACKAGE_ID=...`.

## Receipt Verification

After the wallet returns a transaction digest, verify it through Sui RPC:

```ts
const result = await zkpay.verifySuiPayment({
  intent: payment.intent,
  txDigest: "H2j...",
  coinType: "0x...::usdc::USDC",
  expectedSender: "0x...",
  binding: {
    packageId: "0x..."
  }
});

if (!result.ok) {
  throw new Error(result.errors.join(", "));
}
```

The verifier checks:

- transaction exists on Sui RPC;
- transaction effects succeeded;
- receiver gained the expected coin type;
- receiver amount matches the payment intent after decimal conversion;
- optional sender address had the corresponding debit;
- optional `PaymentBound` event matches payer, receiver, amount, coin type,
  payment id, and nonce;
- generated receipt still passes local intent verification.

## API Boundary

The Hono API exposes a Sui verification route for merchant backends:

```txt
POST /payments/verify/sui
```

Request shape:

```json
{
  "intent": {},
  "txDigest": "H2j...",
  "coinType": "0x...::usdc::USDC",
  "decimals": 6,
  "expectedSender": "0x...",
  "binding": {
    "packageId": "0x..."
  }
}
```

The response includes `ok`, `errors`, a generated `receipt`, local verification
details, transaction deltas, warnings, and replay guard metadata when the
verification succeeds.

## Replay Guard

The Hono API enables an in-process replay store by default. Once a Sui
settlement verifies successfully, the store records both the payment id and
transaction digest.

It rejects:

- the same transaction digest being reused;
- a different transaction digest being submitted for an already settled payment
  id.

Replay failures return `409` with `digest_already_used` or
`payment_already_settled`. The default store is useful for demos and single
process deployments, but production merchant systems should inject a durable
store backed by their database or cache.

## CLI

The CLI can verify a testnet receipt from a checkout URL or serialized intent:

```bash
zkpay receipt verify-sui \
  --intent '<checkout-url-or-json-intent>' \
  --tx-digest H2j... \
  --coin-type 0x...::usdc::USDC \
  --decimals 6 \
  --binding-package-id 0x... \
  --network testnet \
  --json
```

## Merchant Demo

The repository includes a v0.2 merchant example:

```bash
npm run example:merchant
```

It creates a hosted checkout URL with Sui runtime parameters, prints the
backend `/payments/verify/sui` payload, and shows where a merchant would fulfill
after successful verification.

With a real testnet receipt package:

```bash
ZKPAY_BINDING_PACKAGE_ID=0x... npm run example:merchant
```

## Current Boundary

v0.2 verifies settlement by digest, receiver, coin type, amount, replay store,
and optional receipt binding event. The binding package in this repo is still an
alpha Move prototype, not a canonical zkpay mainnet package. Production merchant
systems should deploy or audit their binding package, store verified digests
durably, and reject reuse before fulfillment.
