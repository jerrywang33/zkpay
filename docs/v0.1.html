v0.1 Product Contract

The first zkpay payment model: intent, checkout, gas route, and receipt verification.

Overview v0.1 API Architecture zkLogin Gas routing npm release Roadmap ADR 0001

zkpay starts as a narrow checkout layer for Sui apps:

PaymentIntent -> Checkout Authorization -> Gas Route -> Settlement Receipt

The first useful version is complete when a merchant can create a stablecoin payment link, a payer can authorize the payment, and the merchant backend can verify the receipt before fulfillment.

The current implementation covers the model, SDK, CLI, API boundary, generated checkout URL, and hosted checkout shell. Real Sui transaction building and zkLogin proof execution remain the next integration layer.

Public Alpha

The current public install path is the zkpay-sh npm package:

npm install zkpay-sh@next

It bundles the SDK, core primitives, and CLI:

import { ZkpayClient } from "zkpay-sh";

const zkpay = new ZkpayClient();
const payment = zkpay.createPayment({
  amount: "20",
  coin: "USDC",
  receiver: "0x84f",
  label: "API credits"
});

PaymentIntent

A PaymentIntent is the contract between merchant UI, hosted checkout, and backend verification.

Required fields:

  • id: zkpay payment id, formatted as zkp_*.
  • amount: positive decimal string, never a floating point number.
  • coin: stablecoin symbol or coin type.
  • receiver: Sui receiver address.
  • label: human-readable payment label.
  • metadata: merchant-owned key/value strings.
  • nonce: unique replay-protection value.
  • createdAt: ISO timestamp.
  • expiresAt: optional ISO timestamp.

Gas Route Policy

gasless is a route decision, not a blanket promise.

RoutePayer gasCondition
gasless-stablecoinzeroSupported stablecoin transfer with no extra checkout logic.
sponsoredsponsored by merchant/appCheckout needs receipt or programmable transaction logic.
payer-paidpaid by payerUnsupported route and no sponsor policy is available.

Supported gasless stablecoin symbols tracked by the first model:

USDsui, SuiUSDe, AUSD, FDUSD, USDB, USDC, USDY

Receipt Verification

Merchant fulfillment must verify:

  • receipt status is succeeded;
  • receipt paymentId matches the intent;
  • amount matches exactly;
  • coin matches exactly;
  • receiver matches exactly;
  • nonce matches exactly;
  • transaction digest is present.

The MVP verifier is deterministic and local. Later versions can add Sui RPC lookups, event verification, webhook signatures, and Payment Kit receipt compatibility.

zkLogin Boundary

The first product model reserves the authorization step for zkLogin, but does not make zkLogin mandatory for v0.1. Wallet authorization can validate the payment loop first; zkLogin enters once checkout, gas route policy, and receipt verification are stable.
