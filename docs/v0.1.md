# zkpay v0.1 Product Contract

`zkpay` starts as a narrow checkout layer for Sui apps:

```txt
PaymentIntent -> Checkout Authorization -> Gas Route -> Settlement Receipt
```

The first useful version is complete when a merchant can create a stablecoin
payment link, a payer can authorize the payment, and the merchant backend can
verify the receipt before fulfillment.

The current implementation covers the model, SDK, CLI, API boundary, generated
checkout URL, and hosted checkout shell. Real Sui transaction building and
zkLogin proof execution remain the next integration layer.

## Public Alpha

The current public install path is the `zkpay-sh` npm package:

```bash
npm install zkpay-sh@next
```

It bundles the SDK, core primitives, and CLI:

```ts
import { ZkpayClient } from "zkpay-sh";

const zkpay = new ZkpayClient();
const payment = zkpay.createPayment({
  amount: "20",
  coin: "USDC",
  receiver: "0x84f",
  label: "API credits"
});
```

## PaymentIntent

A `PaymentIntent` is the contract between merchant UI, hosted checkout, and
backend verification.

Required fields:

- `id`: zkpay payment id, formatted as `zkp_*`.
- `amount`: positive decimal string, never a floating point number.
- `coin`: stablecoin symbol or coin type.
- `receiver`: Sui receiver address.
- `label`: human-readable payment label.
- `metadata`: merchant-owned key/value strings.
- `nonce`: unique replay-protection value.
- `createdAt`: ISO timestamp.
- `expiresAt`: optional ISO timestamp.

## Gas Route Policy

`gasless` is a route decision, not a blanket promise.

| Route | Payer gas | Condition |
| --- | --- | --- |
| `gasless-stablecoin` | zero | Supported stablecoin transfer with no extra checkout logic. |
| `sponsored` | sponsored by merchant/app | Checkout needs receipt or programmable transaction logic. |
| `payer-paid` | paid by payer | Unsupported route and no sponsor policy is available. |

Supported gasless stablecoin symbols tracked by the first model:

```txt
USDsui, SuiUSDe, AUSD, FDUSD, USDB, USDC, USDY
```

## Receipt Verification

Merchant fulfillment must verify:

- receipt status is `succeeded`;
- receipt `paymentId` matches the intent;
- amount matches exactly;
- coin matches exactly;
- receiver matches exactly;
- nonce matches exactly;
- transaction digest is present.

The MVP verifier is deterministic and local. Later versions can add Sui RPC
lookups, event verification, webhook signatures, and Payment Kit receipt
compatibility.

## zkLogin Boundary

The first product model reserves the authorization step for zkLogin, but does
not make zkLogin mandatory for v0.1. Wallet authorization can validate the
payment loop first; zkLogin enters once checkout, gas route policy, and receipt
verification are stable.
