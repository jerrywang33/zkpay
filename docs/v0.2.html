v0.2 Testnet Settlement Loop

Build a Sui testnet payment transaction and verify the resulting digest before fulfillment.

Overview v0.1 v0.2 API Architecture zkLogin Gas routing npm release Roadmap ADR 0001

v0.2 moves zkpay from a model-only alpha toward a real Sui testnet payment loop:

PaymentIntent -> wallet transaction -> Sui testnet digest -> RPC receipt verification

The goal is narrow: a merchant can create a payment, a payer can submit a Sui stablecoin transfer, and the merchant can verify the transaction before fulfillment.

SDK Surface

Install the public alpha package:

npm install zkpay-sh@next

Build a wallet-submittable Sui transfer transaction:

import { ZkpayClient } from "zkpay-sh";

const zkpay = new ZkpayClient();
const payment = zkpay.createPayment({
  amount: "20",
  coin: "USDC",
  receiver: "0x...",
  label: "API credits"
});

const { transaction, amountAtomic } = zkpay.buildSuiPaymentTransaction({
  intent: payment.intent,
  payer: "0x...",
  coinType: "0x...::usdc::USDC",
  decimals: 6
});

The returned transaction is a Sui TypeScript SDK Transaction that a wallet adapter can sign and execute on testnet.

Hosted Checkout Handoff

The hosted checkout route now loads a browser wallet runtime:

/pay/:paymentId?intent=...&network=testnet&coinType=0x...::usdc::USDC&decimals=6

The page detects Wallet Standard Sui wallets, asks the payer to connect, submits a stablecoin transfer, captures the returned transaction digest, and then verifies receiver balance changes through Sui RPC.

After a digest is available, checkout renders the backend verification payload:

{
  "intent": {},
  "txDigest": "H2j...",
  "coinType": "0x...::usdc::USDC",
  "decimals": 6,
  "expectedSender": "0x...",
  "amountPolicy": "exact",
  "options": {
    "enforceExpiration": true
  }
}

The wallet handoff is still an alpha settlement path. It does not custody funds and it does not mark merchant orders paid by itself. Merchant backends must call /payments/verify/sui or verifySuiPayment before fulfillment.

Receipt Verification

After the wallet returns a transaction digest, verify it through Sui RPC:

const result = await zkpay.verifySuiPayment({
  intent: payment.intent,
  txDigest: "H2j...",
  coinType: "0x...::usdc::USDC",
  expectedSender: "0x..."
});

if (!result.ok) {
  throw new Error(result.errors.join(", "));
}

The verifier checks:

  • transaction exists on Sui RPC;
  • transaction effects succeeded;
  • receiver gained the expected coin type;
  • receiver amount matches the payment intent after decimal conversion;
  • optional sender address had the corresponding debit;
  • generated receipt still passes local intent verification.

API Boundary

The Hono API exposes a Sui verification route for merchant backends:

POST /payments/verify/sui

Request shape:

{
  "intent": {},
  "txDigest": "H2j...",
  "coinType": "0x...::usdc::USDC",
  "decimals": 6,
  "expectedSender": "0x..."
}

The response includes ok, errors, a generated receipt, local verification details, transaction deltas, warnings, and replay guard metadata when the verification succeeds.

Replay Guard

The Hono API enables an in-process replay store by default. Once a Sui settlement verifies successfully, the store records both the payment id and transaction digest.

It rejects:

  • the same transaction digest being reused;
  • a different transaction digest being submitted for an already settled payment id.

Replay failures return 409 with digest_already_used or payment_already_settled. The default store is useful for demos and single process deployments, but production merchant systems should inject a durable store backed by their database or cache.

CLI

The CLI can verify a testnet receipt from a checkout URL or serialized intent:

zkpay receipt verify-sui \
  --intent '<checkout-url-or-json-intent>' \
  --tx-digest H2j... \
  --coin-type 0x...::usdc::USDC \
  --decimals 6 \
  --network testnet \
  --json

Current Boundary

v0.2 verifies settlement by digest, receiver, coin type, and amount. The payment nonce is still merchant-side state, not an onchain memo or event. The API can reject replay through a merchant-owned store, but a later version should bind the zkpay payment id or nonce onchain through a memo/event/witness contract.
